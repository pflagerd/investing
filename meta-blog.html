I've created a directory called .issues/ in which I want to put code/data/everything pertaining to creating a repository-hosted equivalent to github.com which requires only a browser to access it, but which also has features for command-line (bash) access to the same content in plain text.

I created an issue #10 in github.com/pflagerd/investing. As I add content to it, I'll want to add equivalent content (and tools) to .issues/

I added a css rule for code which defined the font-size to be 120% em - since all the code elements will appear within the <body> element. I used spectacle's rectangle-capture to measure and compare the height of characters displayed herein. The code characters were visibly smaller than the body text characters until I made this change.

I added a folder .issues/10/ to contain the repository-hosted version of issue #10.

I created an html fragment .issues/10/title.html containing the issue's title. Unlike github.com titles, this title contains a hyperlink. If everything is html, then we could have way more interesting features in the repository-hosted issue-tracking system.

I put a screenshot of a youtube.com summary in .issues/10/images/Screenshot_20250512_101928.png

I then adjusted styles.css to make the code style be 1.16em instead of 1.20em, since it looks better in the paragraph immediately above this one. The original size made S.issues/10/images/Screenshot_20250512_101928.png look too big relative to body text. But after fiddling with that and remeasuring, I put it back to 1.20em

If I'm surfing to .issues with bash, I'd like to see a text-only rendering of the contents of the issue when I type ls.

I'm thinking I could achieve this by providing .issues/bin/ls which is aware of relative paths and changes the output accordingly. In this way, ls will appear to behave the same as the stock ls if the current directory needs no unique issue-tracking "overrides". To do this, I'd need some way for the shell session to search the .issues/bin path first.
